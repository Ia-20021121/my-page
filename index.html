01

重大度の判定

まず影響範囲から重大度を決める。判定で対応速度と連絡範囲が変わる。

SEV-1
Critical
一次応答 15分以内
  • 主要サービスが全面停止
  • 広範なユーザー影響・売上影響
  • データ消失・情報漏えいの恐れ
SEV-2
High
1時間以内に着手
  • 一部機能の停止・性能劣化
  • 回避策はあるが影響が継続
  • 一部ユーザーに影響
SEV-3
Low
翌営業日まで対応可
  • 軽微・限定的な不具合
  • 明確な回避策あり
  • ユーザー影響はほぼなし
02

対応フロー

検知から振り返りまでの基本の流れ。SEV-1はステップ4を待たず並行でエスカレーションする。

1

検知

監視アラートまたは問い合わせで覚知。発生時刻を記録し、専用チャンネルを開設する。

記録: 発生時刻 / 検知元 / 初期症状
2

初動・トリアージ

影響範囲（対象サービス・ユーザー数・データ）を素早く確認する。むやみに本番を触らない。

確認: 影響範囲 / 直近の変更・デプロイ
3

重大度の判定

セクション01の基準でSEV-1/2/3を確定。SEV-1なら直ちに次へ。

SEV-1: 判定と同時にエスカレーション開始
4

エスカレーション

インシデント指揮者（IC）を立て、必要な関係者を招集。役割（対応・記録・対外連絡）を分担する。

立てる: IC / 記録係 / 連絡係
5

復旧対応

原因究明より暫定復旧を優先。該当ランブック（セクション04）に沿って手を打つ。実施内容は逐次記録。

優先: 影響の止血 → 暫定復旧
6

復旧確認

監視指標とユーザー影響が解消したか確認。関係者・利用者へ復旧を周知する。

確認: 指標が正常 / 再発なし
7

振り返り（ポストモーテム）

個人を責めない（Blameless）前提で原因と再発防止策を整理。改善タスクは期限つきで起票する。

作成: 時系列 / 根本原因 / 再発防止
03

エスカレーション連絡先

上から順に連絡。15分応答がなければ次の段へ上げる。

役割担当（サンプル）連絡手段担当範囲
一次対応 / オンコール山田 太郎#incident / 内線0000初動・トリアージ
二次 / リードエンジニア佐藤 花子lead@example.com技術判断・復旧主導
インシデント指揮者(IC)鈴木 一郎内線0001全体統括・意思決定
対外・顧客連絡田中 美咲pr@example.com顧客/広報への周知

04

個別ランブック

症状から該当項目を開く。コマンドはダミー。実環境のものに置き換えて使う想定。

SEV-1/2 Webサーバーが応答しない（5xx / タイムアウト）›

症状

  • 監視のHTTPチェックが連続失敗
  • 5xxエラー率の急増・応答遅延

確認

  • プロセス・ヘルスチェックの状態
  • CPU / メモリ / コネクション数
  • 直近のデプロイや設定変更の有無
systemctl status web.service curl -I https://service.example.com/healthz

一次対応

  1. 異常ノードをロードバランサから切り離す
  2. プロセスを再起動して復旧を確認
  3. 直近デプロイが原因ならロールバック

エスカレーション基準

  • 10分で復旧しない、または全ノード影響 → SEV-1としてICへ
SEV-1 データベース接続エラー›

症状

  • "connection refused" / "too many connections"
  • 書き込み系処理が広範に失敗

確認

  • DBプロセスの稼働とディスク空き
  • 接続数が上限に達していないか
  • レプリケーション遅延
SHOW STATUS LIKE 'Threads_connected'; SHOW PROCESSLIST;

一次対応

  1. 長時間ロック・暴走クエリを特定して停止
  2. 接続枯渇ならアプリ側のコネクション設定を見直す
  3. 必要に応じて読み取りをレプリカへ退避

エスカレーション基準

  • プライマリ停止やデータ不整合の恐れ → 即SEV-1・ICとDBA招集
SEV-2 ディスク容量の逼迫›

症状

  • 使用率90%超のアラート
  • 書き込み失敗・ログ出力停止

確認

df -h du -sh /var/log/* | sort -rh | head

一次対応

  1. 肥大化したログ・一時ファイルを退避/削除
  2. ローテーション設定を確認・修正
  3. 恒久対応としてボリューム拡張を起票

エスカレーション基準

  • 削減しても98%超が続く → SEV-1へ引き上げ
SEV-2/3 TLS証明書の期限切れ›

症状

  • ブラウザで証明書エラー
  • API連携がTLS検証で失敗

確認

echo | openssl s_client -connect service.example.com:443 2>/dev/null | openssl x509 -noout -dates

一次対応

  1. 証明書を再発行・更新して反映
  2. 全ノード/ロードバランサへ確実に配布
  3. 自動更新（ACME等）の失敗原因を確認

エスカレーション基準

  • 本番の主要ドメインが影響 → SEV-2、顧客影響が広ければSEV-1
05
